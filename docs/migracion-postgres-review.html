Dejar Supabase → Postgres propio

Review técnico: qué cuesta migrar el backend de Supabase a Postgres corriendo en nuestra propia infraestructura. Escrito sobre el estado real de la app (1,295 líneas de SQL, 18 funciones, 14 vistas, 11 tablas).

En una línea

El motor no cambia: Supabase ya es Postgres. Todo el schema.sql — tablas, funciones, vistas, RLS — se mueve tal cual con un pg_dump, riesgo casi nulo. Lo único que hay que reemplazar son 4 servicios que Supabase pone encima de Postgres y que el front usa: la API REST, el realtime, el login y (a futuro) el storage.

La distinción que lo decide todo

"Migrar a Postgres" suena a cambio de base de datos, pero no lo es. Supabase = Postgres administrado + una capa de servicios. Al irnos, el Postgres se lo lleva completo; lo que se queda en Supabase es esa capa. Y esa capa es open source y autohospedable — se puede correr la misma pieza dentro del banco.

Se mueve gratis sin reescribir

Todo el trabajo pesado de esta sesión. pg_dump | pg_restore.

  • 11 tablas (bronze / silver)
  • 18 funciones plpgsql: el motor load_begin/batch/commit, las 5 guardas, workflow, triage, métricas
  • 14 vistas, incluida la fórmula del risk score
  • 4 políticas RLS + índices + constraints

Hay que reemplazar 4 servicios

Lo que el front consume hoy de la plataforma Supabase.

  • API REST (PostgREST): 14 .rpc + 14 lecturas de vistas
  • Realtime: 3 canales (aviso de carga)
  • Auth (login): 4 llamadas
  • Storage: aún no se usa (el .csv.gz a futuro)

Qué reemplaza a cada servicio

Cada uno de los 4 tiene una versión open source que se autohospeda — es literalmente el mismo software que corre Supabase — o una alternativa estándar.

PostgREST reusable tal cual 28 llamadas del front

Es la API REST que genera los endpoints automáticamente desde el esquema (las v_* y los .rpc()). Es open source y se autohospeda con un binario. Se levanta apuntando al mismo Postgres y el front no cambia una línea — solo la URL base.

Realtime 3 opciones 3 canales

Avisa al front cuando bronze.loads cambia de estado. Opciones: (a) autohospedar el servidor Realtime de Supabase (mismo software); (b) Postgres LISTEN/NOTIFY + un websocket propio — trivial, ya que solo es un evento por carga; (c) o simplemente hacer polling cada N segundos. Con 2 cargas al mes, la opción (c) es más que suficiente.

Auth (GoTrue) oportunidad 4 llamadas

Hoy el login es Supabase Auth. Autohospedable (GoTrue, open source). Pero para un banco lo correcto es integrar el SSO corporativo (Azure AD / LDAP) — algo que de todos modos tocaba hacer para producción. Migrar es la ocasión de hacerlo bien: el usuario entra con su cuenta Scotia, no con una cuenta aparte.

Storage sin impacto hoy

No se usa todavía. Cuando se guarde el .csv.gz de cada carga, cualquier bucket S3-compatible interno sirve.

Dos caminos

A · Autohospedar el stack Supabase completo  recomendado

Correr Postgres + PostgREST + Realtime + GoTrue con Docker, dentro de la red del banco.

  • pg_dump de la base → pg_restore en el Postgres interno. El esquema entra idéntico.
  • Levantar el resto del stack (open source) apuntando a ese Postgres.
  • El front solo cambia la URL de Supabase por la interna. Cero reescritura de código.

Esfuerzo: bajo-medio (infra + Docker). Riesgo al modelo de datos: nulo. Mejor para: un banco que necesita los datos en su red pero no quiere reescribir nada.

B · Postgres pelón + API/auth propios  más control, más trabajo

Solo Postgres. Se construye una API (Node/Go) y se integra auth; realtime por LISTEN/NOTIFY o polling.

  • pg_dump/restore igual — la base entra idéntica.
  • Construir una capa API que exponga las mismas vistas y RPCs (o reescribir el front para hablar con ella).
  • Integrar SSO y el realtime/polling.

Esfuerzo: medio-alto (se construyen 2-3 servicios). Riesgo al modelo: nulo. Mejor para: quien quiera dependencias mínimas y control total del stack, a cambio de mantenerlo.

El ángulo banco

La razón real para dejar Supabase en banca casi nunca es técnica: es residencia de datos. Este dataset — hostnames de PROD, CVEs sin parchar, responsables — probablemente no puede vivir en la nube de un tercero. Eso empuja a autohospedar, y la buena noticia es que autohospedar no obliga a reescribir: el camino A pone exactamente lo mismo que hoy, pero dentro de la red del banco, con SSO corporativo. Es el paso que de todos modos tocaba para producción.

Veredicto

Recomendación

Si el motivo es control, red interna o cumplimiento (lo esperable en banca): camino A — autohospedar el stack Supabase completo. El Postgres se mueve intacto con pg_dump, el front casi no cambia, y ganas residencia de datos + la ocasión de meter SSO. El camino B es válido si quieres dependencias mínimas y no te pesa mantener la API. En ambos, lo más difícil que construimos — el motor de conciliación y la lógica en SQL — se mueve sin tocarse.

Nota: este review es solo Supabase→Postgres. Migrar a MongoDB sería otra historia — perderías las 18 funciones plpgsql y las 14 vistas (el corazón del sistema no tiene equivalente en Mongo) y habría que reescribir el backend. Nuestro dato es profundamente relacional; el motor SQL es el activo. Postgres es la casa correcta.
